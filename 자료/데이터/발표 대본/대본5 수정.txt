저희가 선정한 주제는 졸업요건 조회 사이트입니다.

(페이지 넘김)

저희 시스템의 유스케이스 다이어그램입니다.
관리자와 사용자 두 액터가 있으며,
사용자는 메인화면을 비롯해 졸업요건 조회, 게시판, 교양추천조회를 이용할 수 있고, 관리자는 게시판 글 관리, 교양 과목관리를 관리합니다.

졸업교건 조회는 학업성적확인서 PDF 업로드와 확장관계에 있으며
게시판은 게시판 글 관리를 포함하고 있고
게시판 글 관리로 등록, 수정, 삭제, 조회를 할 수 있다.
또한 제목으로 검색하여 글 조회가 가능합니다.

교양 추천 조회는 교양 과목관리와 포함관계이며
교양 과목관리로 등록, 수정, 삭제를 할 수 있습니다.

(페이지 넘김)

다음은 DB 테이블 입니다.
학과, 학번, 전공필수, 교양, 게시판으로 테이블을 나누어 만들었습니다.
각 학과에 많은 전공필수 과목들이 있어서 학과 테이블에 학과번호를 외래키로 두어서 학과번호에 따라 전공필수 과목들을 알 수 있습니다.
학번마다 교양에 대한 졸업요건들이 차이가 있어서 학번 테이블에 학번컬럼을 외래키로 두어서 학번에 맞게 교양을 알 수 있습니다.

(페이지 넘김)

저희의 주요 기능은 이전 발표에서도 말씀드렸다시피
첫 번째, 졸업 요건 확인 기능
두 번째, 인기 교양 추천 기능
세 번째, 졸업요건 게시판 기능
입니다.

(페이지 넘김)

저희의 세 가지 주요 기능을 상세히 설명하겠습니다.

첫번째, 졸업요건 확인 기능입니다.
먼저 학업성적확인서 PDF 업로드 페이지에서 사용자가 PDF 파일을 업로드합니다.
PDF 파일이 성공적으로 업로드 되면 결과 조회 페이지로 이동하며
업로드가 실패한 경우 또는 PDF 파일이 아닌 파일을 업로드한 경우 또는 학업성적확인서 파일이 아닌 경우에는 안내 문구와 함께 업로드 페이지로 돌아가게 됩니다.

학업성적확인서 PDF 파일이 성공적으로 업로드되어 결과 조회 페이지로 이동한 후,
졸업요건을 만족했을 경우 '졸업 가능!' 페이지를 제공하고
반대로 만족하지 못했을 경우 부족한 졸업요건 목록을 제공합니다.

(페이지 넘김)

그리고 관리자는 PDF 업로드 성공 시
교양 테이블에 해당 교양이 있으면, '카운트' 속성이 1씩 증가하고
만약 해당 교양 과목이 없으면, 관리자에게 알려지게끔 처리합니다.

인기 교양 추천 기능을 위해 교양 카운트 수를 증가해야 하기 때문에 테이블에 없는 교양이 있으면 관리자에게 알려져야 합니다.
관리자에게 알릴 방법은 저희 팀도 구체적으로 정한 바가 없어 프로젝트를 진행하며 고려할 예정입니다.
또한 최대한 테이블에 교양 과목의 누락이 없게끔 노력할 예정입니다.

(페이지 넘김)

두번째, 인기 교양 추천 기능입니다.
관리자가 교양 테이블의 카운트 속성을 내림차순으로 정렬하여 추천 교양 페이지에 제공하여 사용자가 인기 교양을 볼 수 있습니다.

세 번째, 졸업요건 게시판 기능입니다.
저희 게시판 기능에는 조회, 작성, 수정, 삭제, 검색 기능이 있습니다.
사용자가 게시글을 조회하면 테이블의 조회수 속성이 1씩 증가하고
관리자는 게시판 기능을 통해 공지사항과 안내문을 제공합니다.

(페이지 넘김)

게시글 작성 기능은 사용자는 별도의 로그인 없이 제목, 내용, 작성자, 비밀번호로만 게시글을 작성하며 비밀번호는 수정과 삭제를 위해 추가했습니다.
사용자가 게시글 작성을 성공하면, 사용자의 화면은 작성한 게시글 페이지로 이동하고 관리자는 게시글 테이블에 해당 게시글 튜플을 추가합니다.
작성 실패 시 '비어 있는 항목이 있습니다' 문구를 출력하여 사용자가 작성 실패를 알 수 있게끔 하고 사용자가 이어서 작성 할 수 있게 페이지 이동은 없습니다.

(페이지 넘김)

게시글 수정 기능은 작성 시 입력했던 비밀번호와 일치하면 수정 페이지로 넘어가고, 비밀번호가 일치하지 않으면 '비밀번호가 일치하지 않습니다.' 문구와 함께 게시글 페이지로 이동합니다.
수정 성공 시 사용자는 수정한 게시글 페이지로 이동하고 관리자는 게시글 테이블의 수정시간을 업데이트합니다.
수정 실패 시 작성 실패와 같이 '비어 있는 항목이 있습니다' 문구를 출력하며, 작성 실패 시와 마찬가지로 페이지 이동은 없습니다.

(페이지 넘김)

게시글 삭제 기능도 작성 시 입력했던 비밀번호와 일치하면, 비밀번호가 일치하지 않으면 '비밀번호가 일치하지 않습니다.' 문구와 함께 게시글 페이지로 이동합니다.

게시글 삭제 후 게시글 목록 페이지로 이동하고 관리자는 게시글 테이블 해당 튜플의 del_flag를 false에서 true로 업데이트 합니다.

(페이지 넘김)

다음은 게시글 목록 및 검색 기능입니다.

게시글 목록은 게시글 테이블의 del_flag가 false인 튜플만 제공해서 사용자가 게시글 목록을 볼 수 있습니다.

검색 기능은 사용자가 제목이나 내용을 검색하면, 검색어를 포함하는 제목 또는 내용을 게시글 테이블에서 조회합니다.
해당하는 검색어가 있으면 게시글 목록에 제공하고, 없으면 비어 있는 게시글 목록을 제공합니다.

(페이지 넘김)

다음으로 기대 효과입니다.
기대 효과는 기존의 기대효과에서 기능들을 넣었을 때의 기대효과와 인터뷰를 통해 얻을 수 있었던 기대효과를 추가하여 수정했습니다.

첫 번째, 시스템 자동화 입니다. 
사용자가 하나하나 조건을 찾지 않고 학업성적확인서 PDF를 업로드하면 결과 조회 페이지를 보여주고, 사용자의 졸업요건이 미달된 경우 부족한 졸업요건 리스트를 제공함으로써 사용자가 졸업 요건을 수기로 확인하는 것과 비교하여 정확성을 높이고 시간 절약을 할 수 있습니다.

두 번째, 모든 학생이 사용할 수 있습니다.
사용자는 학번, 학년에 상관없이 학업성적확인서 PDF를 업로드하면 부족한 학점과 본인 학번에 맞는 졸업요건 리스트를 제공 받을 수 있고, 인기 있는 교양 추천을 받을 수 있으며 정보 공유 게시판을 이용할 수 있습니다.

(페이지 넘김)

세 번째, 주기적인 방문 효과입니다.
사용자가 졸업 시즌에만 단발성으로 이용하는 것이 아닌 정보 공유 게시판 기능을 통해 다양한 학사정보와 수강신청, 졸업요건 대체 조건 등의 정보를 공유하고 주기적으로 사이트에 방문하게 합니다.

네 번째, 직관전인 UI를 설계해서 사용자가 학업성적확인서 PDF 업로드를 성공한다면 다음 페이지인 결과 조회 페이지의 UI로 사용자의 졸업요건 확인 여부를 쉽게 확인할 수 있도록 합니다.
또한 사용자가 파악하기 쉬운 보기 좋은 구조를 통해 필요한 졸업요건이나 채우지 못한 필수 과목들의 조회 결과를 한눈에 파악할 수 있습니다.

(페이지 넘김)

마지막으로 교무팀과 인터뷰를 진행한 결과, 졸업 시즌에 문의가 집중적으로 발생하고 전체 졸업 예정자 중 37.05%의 학생들이 졸업요건을 채우지 못하고 졸업이 취소되었다는 것을 알 수 있었습니다.
또한 저희 팀이 만드는 졸업요건 확인 시스템을 교무팀에 사용할 의향이 있다는 답변을 받았고,
수기로 작성하여 확인하는 전과 달리 위 시스템이 도입되면 학생들이 본인의 졸업 요건을 자동으로 확인할 수 있고 관리할 수 있어서
"교무팀의 부담을 덜어내고 졸업요건을 채우지 못해 취소되는 경우를 방지할 수 있는 기대효과가 있습니다."
